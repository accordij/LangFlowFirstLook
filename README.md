# LangFlow Workshop Playbook

Единый файл, по которому можно продолжить проект без переписки.

Примеры шаблонов, работающих в развернутом окружении приведены в файле "examples all modules.json"

## Контекст проекта (кратко)

- Стенд: кастомный LangFlow, часть стандартных интеграций может отсутствовать.
- Можно писать кастомные модули.
- Работаем итерациями: каждый шаг отдельным JSON-файлом.
- Кейс: мониторинг кредитных продуктов.
- Базовый банк: Сбер.
- Сравнение: Т-Банк, ВТБ, Альфа-Банк.
- Подача доклада: технично, с умеренным юмором.

## Где лежат шаги

- `steps/00_source_reference.json` - исходный референс.
- `steps/01_minimal_chat_agent.json` - шаг 1.
- `steps/02_add_search_tool.json` - шаг 2.
- `steps/03_search_tuning.json` - шаг 3.
- `steps/04_save_to_file_demo.json` - шаг 4 (демо работы с файлами).
- `steps/05_structured_json_output.json` - шаг 5 (строгий структурированный JSON-вывод).
- `steps/06_llm_generated_langflow_nodes.json` - шаг 6 (генерация корректных нод LangFlow силами LLM + проверка на тестовом полигоне).
- `steps/07_seed_historical_data.json` - шаг 7 (загрузка исторических JSONL-данных из файла в SQLite через шаблонные ноды).
- `steps/08_dual_path_analytics.json` - шаг 8 (сравнение ответов агента: сырая SQL-таблица vs analytics tool).
- `steps/09_dashboard_export.json` - шаг 9, финальный (агент строит графики и экспортирует витрину в HTML).
- `steps/10_dashboard_codegen_export.json` - шаг 10, опционально продвинутый (агент генерирует JS для Plotly, тула валидирует и сохраняет HTML).
- `datasets/historical_credit_snapshots.txt` - базовый синтетический исторический датасет (JSON-строки) для загрузки в БД.
- `datasets/historical_credit_snapshots_480rows.txt` - расширенный датасет для stress-теста LLM (120 snapshot x 4 банка = 480 строк в БД).

- `custom_tools/credit_snapshot_db_tool_prompt.md` - ТЗ-промт для внешней LLM (DeepSeek и аналоги) на генерацию кастомной DB tool.
- `custom_tools/credit_dashboard_code_exec_tool.py` - исходник тулы шага 10 (codegen JS + экспорт HTML); встроен в `steps/10_dashboard_codegen_export.json`, пересборка: `node scripts/build_step10_flow.mjs`.

## Журнал шагов (коротко, по шаблону)

### Шаг 1 - минимальный контур

- Делаем: `Chat Input -> GigaChat Agent -> Chat Output`.
- Ожидаем увидеть: агент отвечает в Playground.
- Файл: `steps/01_minimal_chat_agent.json`.
- Статус теста: `оттестировано, работает`.
- Выводы: базовый запуск на стенде ок, можно наращивать tools.

### Шаг 2 - подключаем первый tool

- Делаем: добавляем `Search Plugin` в порт `tools` агента.
- Ожидаем увидеть: агент сам вызывает tool по запросу поиска условий кредита.
- Файл: `steps/02_add_search_tool.json`.
- Статус теста: `оттестировано, работает`.
- Выводы:
  - tool-call работает;
  - история в рамках сессии есть и пока достаточна;
  - по ВТБ данные находятся лучше, чем по Т-Банк.
- Тестовые запросы:
  - "Найди условия потребительского кредита ВТБ и дай короткую сводку"
  - "Сравни кредит наличными ВТБ и Т-Банк по ключевым условиям"

#### Как работает кастомный tool (инструмент) поиска

- Агент вызывает action `run_search` у `Search Plugin`.
- В вызов передается аргумент `query`.
- Остальные параметры берутся из настроек ноды (`max_results`, `domains`, `safe_search`, `timeout`, `url`, `api_key`).
- Поэтому tool - это обычный скрипт с параметрами, а не семантическая "магия".

### Шаг 5 - структурированный JSON-вывод

- Делаем:
  - усиливаем `system_prompt` агента, чтобы ответ был строго в JSON-формате;
  - добавляем модуль `JSON Validate Notice` между агентом и `Chat Output`;
  - сохраняем результат в файл `credit_report_step5.json`;
  - готовим структуру для следующего шага записи в БД.
- Ожидаем увидеть:
  - в чате и в файле ответ в предсказуемой структуре JSON;
  - сразу после ответа ЛЛМ в чат дописывается проверка: `Json структура валидна` или `Json структура НЕвалидна`;
  - у каждого банка единый набор полей (`bank`, `rate`, `apr_psk`, `max_amount`, `term`, `source_url`, `confidence`, `missing_fields` и т.д.).
- Файл: `steps/05_structured_json_output.json`.
- Статус теста: `оттестировано, работает`.
- Выводы:
  - агент возвращает структурированный JSON в ожидаемом формате;
  - модуль `JSON Validate Notice` валидирует JSON на лету;
  - в чат дописывается результат проверки структуры.
- Тестовый запрос:
  - "Собери и сравни ключевые условия потребительского кредита наличными у Сбера, ВТБ, Т-Банка и Альфа-Банка. Верни строго JSON в согласованной структуре."

### Шаг 6 - генерация корректных нод LangFlow силами LLM

- Делаем:
  - создаем отдельный flow `steps/06_llm_generated_langflow_nodes.json` (шаг 5 не трогаем);
  - возвращаем `GigaChat Agent` и `Search Plugin` в контур;
  - добавляем кастомную tool `Credit Snapshot DB Tool` в `agent.tools`;
  - добавляем тестовый полигон в том же flow: `Text Input -> Credit Snapshot DB Tool Playground -> Chat Output`, чтобы проверять tool отдельно от агента;
  - готовим ТЗ и короткий промт в `custom_tools/credit_snapshot_db_tool_prompt.md`, чтобы генерировать аналогичные ноды внешними LLM;
  - tool принимает `record_json` от ЛЛМ, валидирует структуру и обязательные поля;
  - при валидном JSON tool создает (если нужно) таблицу `credit_product_snapshots` и сохраняет строки в SQLite `sqlite:////tmp/credit_products.db`;
  - при невалидном JSON tool возвращает `status=invalid_json` и список ошибок, чтобы агент поправил payload и вызвал tool снова.
- Ожидаем увидеть:
  - агент умеет работать с двумя инструментами: `Search Plugin` и `save_credit_snapshot`;
  - перед финальным ответом агент вызывает DB tool;
  - в ответе tool есть статус `ok`/`invalid_json`/`error`;
  - при `ok` в БД появляются записи с `snapshot_id` и данными по банкам.
- Файл: `steps/06_llm_generated_langflow_nodes.json`.
- Файл: `custom_tools/credit_snapshot_db_tool_prompt.md` содержит примеры для генерации кастомных нод нейронками.
- Статус теста: `успешно пройдено, протестировано`.
- Базовый тестовый запрос:
  - "Собери и сравни ключевые условия потребительского кредита наличными у Сбера, ВТБ, Т-Банка и Альфа-Банка. Верни итог в JSON и сохрани его через tool."
- Тестовый запрос для проверки SQL подключения
  - SELECT 1 AS ok;
  - SELECT sqlite_version() AS version;
  - SELECT name FROM sqlite_master WHERE type='table';
- Тестовые JSON структуры для проверки механизма валидации можно взять в файле credit_snapshot_db_tool_prompt.md (домашнее задание?: сравнить два промта для генерации ноды, какой даст результат, какой нет, в чем будет разница)

### Шаг 7 - загрузка исторических данных в БД из файла (без ручной вставки JSON)

- Делаем:
  - используем отдельный flow `steps/07_seed_historical_data.json`;
  - кладем исторические данные в `datasets/historical_credit_snapshots.txt` (по одному JSON на строку);
  - строим шаблонный пайплайн `File -> Split Text -> Parser -> Credit Snapshot DB Tool Playground`;
  - `Split Text` делит входной txt на JSON-строки;
  - `Parser` передает строки в `record_json` для DB tool;
  - DB tool валидирует JSON/JSONL и сохраняет строки в SQLite `sqlite:////tmp/credit_products.db`;
  - параллельная SQL-ветка используется только для ручной проверки и очистки.
- Ожидаем увидеть:
  - загрузку исторических снапшотов без ручного копирования JSON в ноды;
  - в логе статус `ok` или `partial_ok`;
  - в SQL-проверке список `snapshot_id` и количество строк по каждому снимку.
- Файл: `steps/07_seed_historical_data.json`.
- Датасет: `datasets/historical_credit_snapshots.txt`.
- Статус теста: `успешно пройдено, протестировано`.
- Фактический результат проверки:
  - `SELECT COUNT(*) AS total_rows, COUNT(DISTINCT generated_at) AS snapshots_count FROM credit_product_snapshots;`
  - получено: `total_rows = 32`, `snapshots_count = 8`.
- SQL для очистки перед повторным сидингом:
  - запускать команды по одной (в некоторых конфигурациях SQL-нода ругается на несколько statement в одном запросе);
  - `DELETE FROM credit_product_snapshots;`
  - `DELETE FROM sqlite_sequence WHERE name = 'credit_product_snapshots';` (опционально, только для сброса счетчика AUTOINCREMENT)
- SQL для проверки:
  - `SELECT snapshot_id, COUNT(*) AS rows_count, MIN(created_at) AS first_created, MAX(created_at) AS last_created FROM credit_product_snapshots GROUP BY snapshot_id ORDER BY last_created DESC;`
  - `SELECT COUNT(*) AS total_rows, COUNT(DISTINCT generated_at) AS snapshots_count FROM credit_product_snapshots;`
  - `SELECT generated_at, COUNT(*) AS rows_per_snapshot FROM credit_product_snapshots GROUP BY generated_at ORDER BY generated_at;`
- Режимы датасета для шага 7:
  - базовый прогон: `datasets/historical_credit_snapshots.txt` -> ожидаемо `total_rows = 32`, `snapshots_count = 8`;
  - stress-прогон для проверки LLM "на прочность": `datasets/historical_credit_snapshots_480rows.txt` -> ожидаемо `total_rows = 480`, `snapshots_count = 120`.

### Шаг 8 - два режима анализа: сырая таблица vs analytics tool

- Делаем:
  - создаем отдельный flow `steps/08_dual_path_analytics.json` на базе шага 6;
  - добавляем к агенту `SQL Database` как tool (`run_sql_query`), чтобы LLM могла смотреть таблицу напрямую;
  - добавляем к агенту `Credit Market Analytics Tool` как отдельную tool (`get_credit_market_analytics`), которая читает те же данные и возвращает агрегированные метрики/факты;
  - оставляем `Search Plugin` и `save_credit_snapshot`, чтобы контур агента оставался совместимым с прошлыми шагами;
  - в `system_prompt` фиксируем режим демонстрации: сначала RAW SQL, затем ANALYTICS по запросу.
- Ожидаем увидеть:
  - агент может ответить только на базе `run_sql_query` (сырые строки, слабее структурированные выводы);
  - агент может повторно ответить через `get_credit_market_analytics` (более четкие тренды и ссылки на факты/дельты);
  - на одинаковом вопросе видно влияние предварительной аналитики на качество reasoning.
- Файл: `steps/08_dual_path_analytics.json`.
- Статус теста: `подготовлено для демонстрации`.
- Тестовые запросы:
  - "Сделай вывод по динамике Сбер, ВТБ, Т-Банк и Альфа-Банк, только по сырым данным из таблицы. Используй только run_sql_query, не вызывай get_credit_market_analytics. Дай короткий вывод по каждому банку и общий вывод по рынку."
  - "Сделай вывод по динамике Сбер, ВТБ, Т-Банк и Альфа-Банка. Дай короткий вывод по каждому банку и общий вывод по рынку. Обязательно вызови get_credit_market_analytics. Приведи минимум 3 факта с цифрами из facts"

  - создаем отдельный flow `steps/09_dashboard_export.json` на базе шага 8;

### Шаг 10 (опционально) — дашборд с кодогенерацией графика

- Делаем:
  - отдельный flow `steps/10_dashboard_codegen_export.json` на базе шага 9;
  - в `system prompt` зафиксирована последовательность: перед генерацией `chart_code` агент обязан вызвать `run_sql_query` для `PRAGMA table_info(credit_product_snapshots)` и `SELECT * FROM credit_product_snapshots ORDER BY created_at DESC LIMIT 12`;
  - нода `Credit Dashboard Code Exec Tool`: агент передает в tool аргумент `chart_code` (JavaScript с `async function renderDashboard(payload, Plotly, containers)`), тула валидирует код, собирает HTML, сохраняет в `/tmp/<flow_id>/`, загружает в Files;
  - payload для рендера расширен: помимо совместимых `dates/traces_rate/traces_amount/summary`, tool передает `columns`, `records`, `sample_rows`, `profile`, `suggested_charts` — чтобы агент мог строить график под фактическую структуру таблицы, а не под заранее зашитый шаблон.
- Ожидаем увидеть:
  - по запросу дашборда агент сначала читает схему/сэмпл SQL-таблицы, затем вызывает tool с полным `chart_code`, а в чате — краткий итог и `file_path`;
  - у разных запусков визуализация различается (код пишет модель) и привязана к тем колонкам, которые реально есть в таблице.
  - защита от пустого дашборда: в инструкциях и описании tool закреплено требование минимум `2` вызова `Plotly.newPlot` + fallback на `summary/sample_rows`, если основные traces пустые.
- Файл: `steps/10_dashboard_codegen_export.json`.
- Статус теста: `подготовлено для демонстрации` (проверьте на стенде с установленным `node` для `node --check`, иначе проверка синтаксиса JS смягчается).
- Тестовый запрос:
  - "Собери HTML-дашборд по данным из БД: три графика в контейнерах chartA/chartB/chartC, не дублируй JS в чат. Сохрани как step10_demo.html."
